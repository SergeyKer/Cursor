# Аудит режима «Перевод» (карточки, success/error, задержки)

**Дата:** 2026-04-15  
**Область:** [`components/Chat.tsx`](../components/Chat.tsx) (UI, `parseTranslationCoachBlocks`, `buildAssistantSections`, `MessageBubble`), [`app/api/chat/route.ts`](../app/api/chat/route.ts) (постобработка перевода), [`lib/translationGoldEnglish.ts`](../lib/translationGoldEnglish.ts).  
**Ограничение:** изменений в код не вносилось — только анализ и рекомендации.

## 1. Что сделано автоматически

- Прогнаны существующие тесты Vitest (связанные с переводом): **75 тестов, 0 падений**  
  (`translationProtocolStatus`, `Chat.translationComment`, `translationCommentCoach`, `translationMode`).
- Полный ручной прогон **10× (4 ошибки + 1 успех)** в UI в этой сессии **не выполнялся** (нет доступа к вашей живой сессии/браузеру). Ниже — чеклист и ожидаемое поведение по коду для самостоятельной фиксации фактов.

## 2. Чеклист ручного прогона (до 10 ситуаций)

Для **каждой** ситуации: одна и та же русская фраза из drill на шагах 1–5.

| Шаг | Пользователь | Проверить в UI |
|-----|--------------|----------------|
| 1–4 | Неверный перевод (латиница, по возможности разные типы ошибок) | Ветка ошибки: 💡 из **Комментарий_перевод** (поддержка, не «успешная» похвала); блок **Ошибки** при наличии; **одна** зелёная «Скажи»; нет новой «Переведи далее» с **другой** фразой до успеха; заголовок скрытого повтора — «Переведи», не «Переведи далее» (см. комментарий в `MessageBubble`). |
| 5 | Верный ответ | ✅ похвала по **Комментарий** конкретная; «Переведи далее» с **новой** русской фразой; нет лишнего «Скажи» для той же drill-фразы. |

**Фиксировать:** скрин или полный текст ответа ассистента, несоответствие ветки, дубли карточек, нерелевантность 💡/✅ к последнему ответу.

## 3. Находки и оценка (по статическому анализу + тестам)

Формат: **ID**, симптом, слой, **P-критичность**, оценка частоты, причина, рекомендация, трудозатраты.

### AUD-01 — Дополнительный вызов LLM на эталонный перевод RU→EN

- **Симптом:** ответ пользователю приходит заметно позже, чем «один чат-запрос»; в логах/трассах возможен второй вызов провайдера.
- **Слой:** [`route.ts`](../app/api/chat/route.ts) → `resolveGoldTranslation` / [`finalizeTranslationResponsePayload`](../app/api/chat/route.ts) → [`translateRussianPromptToGoldEnglish`](../lib/translationGoldEnglish.ts) (`callProviderChat`, `maxTokens: 64`).
- **Критичность:** **P2** (latency/стоимость, не обязательно видимый «баг»).
- **Частота:** на ход перевода, когда нет готового скрытого эталона в предыдущей карточке; в рамках **одного** `POST` есть кеш `translationGoldCache` — повтор для той же RU-строки не бьёт API второй раз.
- **Причина:** золотой эталон нужен для `computeTranslationGoldVerdict`, clamp «Скажи», скрытая строка `__TRAN__` / canonical ref.
- **Рекомендация:** оставить кеш между **запросами** (TTL/ключ `ru+level+audience`) при контроле стоимости; либо чаще пробрасывать эталон из предыдущего ответа, чтобы реже вызывать gold. **Оценка: M.**

### AUD-02 — Длинная цепочка синхронной постобработки после основной модели

- **Симптом:** задержка до JSON-ответа API после получения текста от провайдера; нагрузка на CPU.
- **Слой:** [`route.ts`](../app/api/chat/route.ts) (фрагмент ~7441–7537): `normalizeTranslationCommentStyle`, `ensureTranslationProtocolBlocks` / `ensureTranslationSuccessBlocks`, `enrichTranslationCommentQuality`, `replaceFalsePositiveTranslationErrorWithPraise`, `sanitizeTranslationPayloadContinuousErrors`, `applyCefrOutputGuard`, `applyTranslationCommentCoachVoice`, `injectSentenceTypePopravImperative`, `finalizeTranslationResponsePayload` (enforce repeat, sanitize, при необходимости gold), и т.д.
- **Критичность:** **P2** (latency без второго LLM).
- **Частота:** каждый успешный проход валидации `isValidTutorOutput` в режиме translation.
- **Причина:** много узкоспециализированных проходов по одной строке.
- **Рекомендация:** профилировать один реальный ответ; объединить совместимые преобразования в один проход / конвейер с явными стадиями; убрать no-op ветки. **Оценка: L.**

### AUD-03 — Объём системного промпта режима translation

- **Симптом:** выше TTFT и стоимость основного запроса; модель чаще «теряется» в длинных правилах.
- **Слой:** [`route.ts`](../app/api/chat/route.ts) — сборка промпта при `mode === 'translation'` (блоки контракта drill, SUCCESS/ERROR протокол, тон, emoji, C1/C2 и т.д.).
- **Критичность:** **P2** (скорость + качество).
- **Частота:** каждый ход.
- **Рекомендация:** сжать дубли с общими правилами диалога; вынести «контракт вывода» в короткий нумерованный список; A/B по длине. **Оценка: M–L.**

### AUD-04 — Дублирование логики «успех / ошибка» на сервере и в клиенте

- **Симптом:** редкие рассинхроны: бэкенд переписал текст, фронт классифицирует ветку иначе; другая карточка, чем ожидал протокол.
- **Слой:** сервер — `ensureTranslationProtocolBlocks`, `translationSuccessShape`, `canTreatTranslationAsSuccess`, gold-verdict; клиент — [`translationResponseHasSuccessShape`](../components/Chat.tsx), `translationErrorCoachUi`, [`resolveTranslationProtocolStatus`](../lib/translationProtocolStatus.ts).
- **Критичность:** **P1** при проявлении; **P3** как риск до первого кейса.
- **Частота:** низкая, при «кривом» или граничном выводе модели до нормализации.
- **Рекомендация:** единый парсер протокола в `lib` (общий для API нормализации и UI) + тесты на одни и те же фикстуры. **Оценка: L.**

### AUD-05 — Фолбэк «Комментарий_перевод» при отсутствии у модели

- **Симптом:** 💡 с **общим** текстом («Есть хорошая основа…»), слабо привязанным к конкретной ошибке пользователя.
- **Слой:** [`ensureTranslationProtocolBlocks`](../app/api/chat/route.ts) (~3340–3344).
- **Критичность:** **P2** (качество UX, не ломка протокола).
- **Частота:** когда модель не выдала `Комментарий_перевод:`.
- **Рекомендация:** ужесточить промпт / пост-проверку; короткий repair-вызов только на строку поддержки (дорого — взвесить). **Оценка: S–M.**

### AUD-06 — Сервер может перевести «модельно успех» в ошибку (gold / mismatch / prior repeat)

- **Симптом:** пользователь видит ERROR и «Скажи», хотя формулировка ответа казалась правильной; или наоборот — комментарий модели не совпадает с финальной веткой.
- **Слой:** `canTreatTranslationAsSuccess`, `translationGoldVerdictFailed`, `forceTranslationWordErrorProtocol`, сравнение с `priorAssistantRepeatEnglish`.
- **Критичность:** **P1** при жалобах пользователей.
- **Частота:** средняя в пограничных переводах (синонимы, опускание наречий и т.д., если вердикт слибо жёсткий).
- **Рекомендация:** собрать реальные отрицательные примеры из чеклиста §2; калибровать `computeTranslationGoldVerdict` и пороги; логировать `translationGoldVerdictFailed` + user text (без PII в проде — хэш). **Оценка: M.**

### AUD-07 — Потенциал дубля строк «Скажи» в сыром `content` после нормализации

- **Слой:** хвост [`ensureTranslationProtocolBlocks`](../app/api/chat/route.ts) (~3376–3382): при определённых комбинациях `repeat` / `repeatRu` возможна двойная запись в текст (клиент частично компенсирует через `hideEnglishRepeatCard`).
- **Критичность:** **P2**, если парсер на клиенте отобразит два блока или озвучка/копирование запутают.
- **Частота:** низкая; требует подтверждения ручным прогоном §2.
- **Рекомендация:** инвариант «в финальном content не более одной канонической англ. эталонной строки для карточки» + фикстура в тестах (когда будет разрешено менять код). **Оценка: S.**

## 4. Сводка по задержкам и «лишнему» алгоритму

| Узкое место | Влияние | Рекомендация (кратко) |
|-------------|---------|------------------------|
| Второй LLM (`translateRussianPromptToGoldEnglish`) | +1 round-trip к провайдеру, кеш только внутри одного POST | Межзапросный кеш или чаще использовать скрытый ref из прошлого хода |
| Много последовательных `repaired = f(repaired)` | CPU, время до ответа клиенту | Профилировать, слить проходы |
| Длинный translation system prompt | TTFT, токены, ошибки формата | Сжать и дедуплицировать правила |
| Дублирование парсинга server/client | Риск багов, не скорость | Общий `lib`-парсер |
