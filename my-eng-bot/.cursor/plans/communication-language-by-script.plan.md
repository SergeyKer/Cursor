# Общение: язык по «раскладке» текста (кириллица / латиница)

## Ваша логика

- В сообщении есть **кириллица** (как будто осталась **русская раскладка**) → вести диалог **по-русски**.
- В сообщении **только латиница** (английская раскладка) → **английский**.
- Реальная физическая раскладка клавиатуры в веб-приложении недоступна; прокси — **какие буквы попали в строку**.

## Сверка с текущей реализацией

| Место | Правило | Статус |
|--------|---------|--------|
| Клиент `speechLocaleForCommunication` в [`components/Chat.tsx`](../components/Chat.tsx) | кириллица → `ru-RU`, только латиница → `en-US`, пусто/только цифры/знаки → `ru-RU` | Уже совпадает с вашей логикой |
| Сервер `detectedUserLang` для `communication` в [`app/api/chat/route.ts`](../app/api/chat/route.ts) | сейчас `detectLangFromText` (подсчёт букв и смешанных слов) | **Расхождение**: смесь «прайс Porsche…» даёт `en` |

## План работ

1. Ввести на сервере функцию **`detectCommunicationUserMessageLang(text, tieBreak)`** с тем же смыслом, что и диктовка: любая кириллица → `ru`; иначе, если есть латиница → `en`; иначе `tieBreak` (например последний язык ассистента / `ru`).
2. В вычислении `detectedUserLang` для режима `communication` (когда это не только «Подробнее») использовать **её** вместо `detectLangFromText(communicationLanguageProbe, …)`.
3. Не менять `detectLangFromText` для **диалога** и **перевода** — только ветка `communication`.
4. Прогнать `npx tsc --noEmit` и при необходимости `npm run lint`.

## Заметки

- `communicationLanguageHint` в промпте может по-прежнему опираться на последний ответ ассистента; целевой язык ответа и fallback’и привязаны к `detectedUserLang` — это главное для фикса «прайс Porsche…».
- Пользователь, пишущий **только латиницей** в общении, получит английский — по вашему правилу.
