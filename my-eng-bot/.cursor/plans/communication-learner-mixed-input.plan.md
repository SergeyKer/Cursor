# Общение: смешанный ввод (ученик не знает слова на английском)

## Сценарий

Пользователь **не знает перевода** и пишет «как получится»: одно предложение с задумкой **на английском**, но **часть слов по-русски** (например `I люблю маму`, `I love мама`).

**Ожидаемое поведение ИИ**

- Воспринимать это как **одну реплику с намерением сказать полное предложение на английском**, где русские слова — **замена неизвестных** английских слов, а не отдельный «переключение языка» в середине фразы.
- **Не** уходить в режим репетитора (без «переведи», «исправь», протокольных блоков — уже запрещено в communication).
- Ответ: **коротко показать понимание** (например: «Да, я понимаю, ты хочешь сказать, что любишь маму» / по тону аудитории), **естественно продолжить диалог** (вопрос или реплика в тему), без длинной лекции.

## Связь с детекцией языка в коде

- При **наличии кириллицы** текущая логика `detectCommunicationUserMessageLang` даёт **`ru`** → постобработка и ориентация ответа на **русский** согласуются с ответом-подтверждением по-русски.
- Правила в **system prompt** сейчас говорят про «majority Cyrillic vs Latin» (текст промпта), а сервер для communication использует **«любая кириллица → ru»** — при правке промпта **выровнять формулировку** с фактической логикой, чтобы модель не получала противоречие.

## Реализация (после согласования плана)

1. В [`app/api/chat/route.ts`](../../app/api/chat/route.ts) в блок `mode === 'communication'` внутри `buildSystemPrompt` добавить явный подпункт:
   - смесь латиницы и кириллицы в **одном** пользовательском сообщении → трактовать как **замещение неизвестных слов** в задуманной англоязычной фразе;
   - отвечать **на языке текущего диалога** (как сейчас), **подтвердить понимание** одной-двумя фразами при необходимости, **продолжить беседу**;
   - не исправлять пользователя как на уроке, если он этого не просил.

2. При необходимости слегка подправить строки про «Language detection» в том же блоке, чтобы они совпадали с `detectCommunicationUserMessageLang` / `getExpectedCommunicationReplyLang`.

3. Ручная проверка: `I love мама`, `I люблю маму`, короткий ответ с пониманием + продолжение; убедиться, что нет лишнего «переведи на английский».

## Полностью русская фраза (кириллица)

- Сообщение **целиком в кириллице** (по правилам детекции в коде: есть кириллица → язык реплики **русский**).
- **Авто-переключение на русский:** контекст ответа и ожидаемый язык реплики — **русский**; при наличии UI-кнопки ожидания ввода — **синхронизировать** `communicationInputExpectedLang` на **`ru`** после отправки (или при отображении), чтобы не держать «ожидание EN», когда пользователь уже пишет по-русски.

## Доп. сценарии: только перевод (без болталки)

**Важно:** режим «**только перевод этой фразы, без комментариев**» включается **только** при **явной** просьбе перевести. Автоматически по одной латинице («английская раскладка») **не** включать.

### Явная просьба «переведи» + фраза (русская или латиница под смысл)

- Пользователь **явно** просит перевести (например «переведи», «translate», «translation», сочетания вроде «как будет по-английски» — по согласованному списку в промпте) и даёт **фразу для перевода**.
- **Ответ ИИ:** **только** английский перевод этой фразы (по смыслу); **никаких** доп. комментариев, похвалы, вопросов и приглашений продолжить диалог.

### Русский текст, набранный в английской раскладке (только латиница), **без** слова «переведи»

- Ввод **только латиница**, по смыслу возможна русская фраза с неверной раскладкой.
- **Не** считать это автоматически режимом «только перевод». Ответ — **обычный чат**: кратко понять намерение, при необходимости уточнить, продолжить диалог — **как в основном сценарии**, пока пользователь **сам** не попросит перевод явно.

### Приоритет правил

1. Сначала проверка: **явная команда перевода** → **строго только EN-перевод** фразы.
2. Иначе полностью кириллица → **русский** контекст (+ авто-русское ожидание ввода, если кнопка есть).
3. Иначе смесь кириллицы и латиницы как «англ. фраза с русскими подстановками» → понимание + продолжение диалога.
4. Иначе только латиница (в т.ч. «под русскую раскладку») **без** явной команды перевода → не translate-only, а обычное общение.

### Краткий алгоритм

1. Есть **явные** маркеры «переведи» / translate + фраза → **только** EN-перевод.
2. Иначе реплика **целиком на кириллице** → русский, авто-ожидание ввода **ru** (кнопка).
3. Иначе смесь латиницы и кириллицы (`I love мама`, `I люблю маму`) → сценарий ученика + продолжение диалога.
4. Иначе только латиница без явной команды перевода → **не** translate-only; обычный ответ в чате.

### Реализация

- В `buildSystemPrompt` для `communication` **переписать** блоки translate-only: убрать автоматический translate-only для «wrong keyboard layout»; оставить жёсткий **translate-only** только при явной команде.
- Опционально: на клиенте после отправки сообщения, если весь текст на кириллице, выставлять `communicationInputExpectedLang = 'ru'`.
- Серверный precheck для translate-only — **только** при детекции явной команды перевода (регулярки/ключевые слова), не для латиницы вообще.
